SECTION 01

The problem: a check-engine light for a camera

A camera is bolted to the outside of a vehicle or robot. The systems downstream — driving, navigation, recording — blindly trust whatever image arrives. But outside, in the weather, the lens fogs up, frosts over, gets splattered with mud, or is partly blocked by ice or tape. A blurry feed that nobody flags is dangerous: the rest of the system keeps trusting garbage.

Our module is the camera's check-engine light. It watches the feed and answers one question every tenth of a second: "Can this picture be trusted, and if not, roughly why?" It puts the answer into one of five labels:

CLEAN

The lens is fine. Trust the image.

FOG

Atmospheric haze — the whole frame goes grey and low-contrast.

FROST

Ice or condensation coating the lens — bright, uniform, and severely blurred.

OBSTRUCTION

Something stuck on the lens in one spot — mud, a leaf, tape, ice.

UNKNOWN

Reserved for "the signals contradict each other." (Defined, but the current code never needs it.)

The deliberate scope limit The detector only labels degradation — it never tries to repair the image (no de-fogging, no de-blurring). Restoration is a research-grade problem that needs gigabytes and a GPU. Detection is cheap, useful, and fits on a chip. Narrow scope is what makes the whole thing possible. One degradation — water droplets / smear — is also deliberately excluded everywhere, because it cannot be detected reliably within this budget, and a half-working safety feature is worse than none.
SECTION 02

The hardware that dictates every rule

Here is the single most important idea in the whole project: almost every coding rule comes from one fact about the chip. Learn the chip, and the rulebook stops feeling arbitrary — you can re-derive it yourself.

What is an ARM Cortex-M0?

Not the laptop you compile on. Picture a whole tiny computer on a single chip — a microcontroller (MCU) — living inside the camera's image-signal processor. It has:

  • No FPU (floating-point unit) — literally no circuitry for decimal math.
  • Kilobytes of RAM, not gigabytes. Our entire budget is under 2 KB.
  • A slow core clocked in tens of MHz, with no cache, shared with other jobs.
  • It must keep up with live video — roughly 10 frames per second.

Deriving the entire rulebook from that one fact

Hardware reality⟶ Rule it forcesWhy
No FPUNo float / double — anywhere Each float op would be faked by a slow software routine (__aeabi_fmul…). On live video that is unaffordable.
Tiny RAM, no heapNo malloc/free; static only; all sizes compile-time constants You must prove RAM use stays under budget before the device runs. A heap can fragment and crash.
Slow, cacheless coreSingle pass over the image at stride 4 Looping over memory is the dominant cost. One pass (not six) and skipping pixels = ~96× less work.
No hardware divideBit-shifts & fixed-point instead of division x >> 4 divides by 16 in one cheap instruction; real division is emulated and slow.
No OS / consoleNo printf, no assert in the core; explicit return codes There is nowhere to print and nothing safe to abort to in field firmware.
One source, two targetsNo #ifdef — pure portable C99 The same degradation.c compiles for the PC test build and the ARM chip, so PC tests validate the shipped code.
The refrain to remember No FPU → no floats. Tiny RAM → static only. Slow core → single pass + stride 4. No divide → shifts. Every strange-looking line later in the code is one of these four trades.

The memory budget, made tangible

The one piece of state the detector keeps between frames is a single struct, DegradationState, totalling 230 bytes. Here is where every byte goes inside the 2 KB ceiling:

SECTION 03

The integer / fixed-point toolkit

If we can't use decimals, how do we compute an average, a contrast ratio, or a percentage? With a handful of integer tricks that show up everywhere in the code. Learn them once here and the rest reads easily.

① Fixed-point: fake decimals with integers

To represent 0.7 without a float, multiply by a scale first. Using 256 as "1.0":

// want 0.7, but no floats allowed
0.7 × 256 ≈ 179     // store 179, meaning "179/256"

That's why you'll see contrast computed as (max-min) * 256 / (max+min+1) — multiply before dividing so the fraction survives as an integer 0–255.

② Shift instead of divide

Dividing by a power of two is just sliding the bits right:

x >> 5  ==  x / 32     (2⁵ = 32)
x >> 4  ==  x / 16
x >> 8  ==  x / 256

One instruction, no divide unit needed. The histogram bins a pixel with pixel >> 5 (8 buckets); the EMAs divide by 16 and 256 with >>4 and >>8.

③ The +1 divide-by-zero guard

An all-black frame has max = min = 0, so a contrast denominator could be zero. Adding +1 makes it harmless:

(max - min) * 256 / (max + min + 1)

It's a deliberate safety guard, not a rounding error. You'll see the same +1 in the ratio helper.

④ Online variance: E[x²] − E[x]²

Variance normally needs the mean first (two passes). The algebra lets us do it in one pass by keeping a running sum and sum-of-squares:

var = sumsq/n − (sum/n)²

Integer flooring can make this dip slightly negative even though true variance ≥ 0 — so the code clamps the negative case to 0.

Overflow is a real enemy — and it's documented at every step Integers wrap around silently when they get too big. Disciplined embedded code proves it can't happen. Every accumulator in the code carries a comment bounding its worst case against its ceiling. The most elegant example: the slow EMA stores values up to 255 × 256 = 65 280, which is just 255 short of the uint16_t ceiling of 65 535 — uint16_t is the smallest type that fits, chosen on purpose.
SECTION 04

Turning pixels into 9 numbers — in one pass

A 320×240 frame is 76,800 pixels. We can't reason about 76,800 numbers ten times a second. So the first job is to boil each frame down to ~9 small numbers ("features") that summarise it — and do it as cheaply as possible.

The two performance tricks, visualised

We (a) walk the image only once, updating every metric as we go, and (b) look at only every 4th pixel in each direction (stride 4 = 1/16 of pixels). Toggle below to feel the difference:

Edge-safe loop bounds The sharpness metric needs each pixel's 4 neighbours. Rather than test "am I on the edge?" on every single pixel (a branch that costs cycles 4,560 times a frame), the loop simply starts at 1 and stops one pixel early. The 1-pixel border is never sampled — under 1% of the image — and neighbour reads are always safe. A classic embedded trade: give up a negligible amount of data to delete a per-pixel branch.

Computed ≠ used — a crucial distinction The detector computes 9 features, but the classifier only reads 6 of them. occlusion_score is kept purely as a diagnostic (it shades the dark-blob cells in the live viewer); the two temporal features are reported for logging. A beginner naturally assumes "everything computed feeds the decision" — here it doesn't, and knowing that prevents a real misconception.

The spatial grid: how we tell local from global

The frame is divided into a 4×3 grid of 12 cells. For each cell we keep a sum and sum-of-squares (so we get its mean and variance in one pass). This is the single most important structural idea in the feature set: fog and frost degrade every cell equally, but a mud blob makes some cells look completely different from others. SECTION 05

The fingerprints: why the features separate the classes

Here's the catch that drives the whole design: fog, frost, and a blocked lens all reduce sharpness. No single number can tell them apart. But each cause leaves a unique combination — like a doctor reading temperature and blood pressure and symptoms together, never one alone.

CLEAN — trust it
sharp ✔ · good contrast ✔ · wide histogram ✔ · uniform
FOG — grey veil, everywhere
contrast <40 · blur 5–15 · mid-tones · uniform · lit
FROST — bright, blurred coat
blur <5 (near zero!) · very bright · uniform
OBSTRUCTION — local blob
some flat cells · very non-uniform
The cleverest insight in the project: blur < 5 vs blur < 15 On-lens coatings vs. atmospheric haze. Ice or condensation on the glass destroys all fine detail — the Laplacian collapses to near zero (< 5). Fog is in the air; it softens the image but detail survives a bit (5–15). That one threshold split — where the blur is coming from — is what cleanly separates FROST from FOG.
The dark-scene guard (a micro-lesson worth its own line) A pitch-black night scene satisfies every fog condition: low contrast ✔, low spread ✔, low sharpness ✔, uniform ✔. Without a guard it would be falsely called FOG. So FOG additionally requires global_mean > 40 — "there has to be some light." This is exactly why the test suite includes a 15-frame pitch-dark segment: to prove darkness stays CLEAN.
SECTION 06

The decision — try it yourself

The classifier is a simple if / else-if ladder tested in strict priority order: OBSTRUCTION → FROST → FOG → CLEAN. First match wins. No machine learning, no baseline — just fixed numeric cutoffs on this frame's raw features. Drag the sliders (or pick a scenario) and watch which branch fires and why.

Looks like a bug — but it's intentional The FROST branch fires when sharpness < 5, but computes its confidence as (15 − sharpness) × 10 — using the other threshold (15). So the instant FROST fires, confidence is already ≥ 110. That's deliberate: a confirmed on-lens coating is a strong, high-confidence call. We flag it here so you don't read it as an error.

SECTION 07

Making it stable over time

A per-frame verdict alone would be jittery and noisy. Two temporal mechanisms turn a math demo into reliable firmware: smoothed baselines (EMA) and hysteresis.

Exponential moving averages — a running average that gently forgets

An EMA nudges a stored average a little toward each new reading. The whole update is one line, and — because the fraction is a power of two — it uses only a shift and adds, no division:

ema = ema − (ema >> shift) + raw;   // shift 4 → α=1/16 (fast),  shift 8 → α=1/256 (slow)

We keep two timescales. The fast EMA reacts in ~16 frames; the slow EMA barely moves over ~256 frames — a stable "what clean looks like" reference. Feed a signal below and watch them track:

raw feature (noisy) fast EMA · α=1/16 slow EMA · α=1/256
Why freeze the slow EMA during a fault? If the "clean reference" kept absorbing frames during a long fog event, it would slowly adapt to the fog and forget what clean ever looked like. So the slow EMA is frozen whenever a degradation is confirmed. Toggle the checkbox above to see it flatline through the event instead of drifting upward.

A subtlety worth knowing: in today's code the classifier is purely absolute-threshold, so these EMAs are maintained but not actually read by the decision. They back the motion metric and the save-to-flash feature, and are kept ready for a future "relative" mode.

Hysteresis — the thermostat principle

A thermostat doesn't flip the heater on and off every few seconds. Likewise, we demand sustained evidence before changing the reported state. It is deliberately asymmetric:

  • 5 consecutive degraded frames to raise an alarm (slow to cry wolf).
  • 15 consecutive clean frames to clear it (even slower to declare "all fixed").
  • Switching between two different faults (e.g. frost → obstruction) happens immediately.

Warmup is advisory, not a gate There's a 60-frame (~6 s) warmup so the slow baseline can settle. But the classifier runs from frame 1 — warmup just sets a flag a host app may use to ignore early output. The library never suppresses anything itself.
SECTION 08

Spec vs shipped code — the honest divergence

This is the most valuable real-world lesson in the project: the design evolved away from its original specification, and the code — not the spec — is the source of truth. The brief (project_brief.md / CLAUDE.md) describes one design; degradation.c ships a simpler, more robust one. A disciplined project keeps a reference oracle and an equivalence test precisely so this drift is caught and controlled.

📄 What the brief describes

  • A baseline-relative ratio classifier: ratio = current·128 / (baseline+1)
  • 6 classes, including a separate DIRT and an emitted UNKNOWN
  • Decisions made by comparing each feature to its learned EMA baseline

✅ What the code actually ships

  • Absolute thresholds — no ratio code exists in degradation.c at all
  • 5 classes defined, 4 emitted: DIRT folded into OBSTRUCTION; UNKNOWN never returned
  • The EMA baseline is computed every frame but the decision tree never reads it
Why DIRT was merged into OBSTRUCTION Think about the operator's response. A dark blob (mud) and a bright blob (tape, ice) lead to the same action: clean the lens. So the two collapse into one OBSTRUCTION verdict. The dark-blob signal is still surfaced separately as occlusion_score for diagnostics — useful to a human, but it doesn't change the decision.
Why absolute thresholds won Absolute cutoffs are far easier to tune (expose the lens to fog, watch the raw number, set the line) and they make exact Python⇄C equivalence trivial to verify — there's no warmup-convergence state to reproduce. The ratio machinery wasn't deleted; it was demoted to dormant, ready for a future relative mode. The takeaway: when the docs and the code disagree, trust the code — and keep a test that proves which is true.
SECTION 09

Proving it: tune in Python, ship in C

How do you trust hand-written integer firmware? You keep a second implementation in Python — detector.py — that is a bit-for-bit integer port of the C, and you prove they produce identical output on every frame.

The oracle: detector.py

Not a loose prototype — a faithful port. It uses NumPy only to sample pixels; all the arithmetic uses // (floor division) and >> to mimic C integer truncation exactly. Every threshold constant is duplicated verbatim from degradation.h. You tune features here (fast, inspectable), then mirror them into C.

The proof: equiv_check.py

It builds a deterministic 165-frame synthetic stream, runs it through both the Python oracle and the compiled C harness, and compares all 12 output columns on every frame as strings — the strictest possible check. Any rounding, shift, or overflow difference shows up instantly.

The synthetic test stream

Each segment is hand-crafted to push one feature into one threshold band, so each provably exercises one class (e.g. the fog frame uses a tiny checker pattern tuned so sharpness lands at ~9 — inside the 5–15 fog band, not the <5 frost band). Press run to scan the equivalence check:

165 frames × 12 columns — ready
▬ clean ▬ fog ▬ frost ▬ obstruction ▬ dark→clean
Why exact equality is even possible Bit-identical agreement across two languages is only achievable because the algorithm is pure integer math. The instant a single float entered, rounding would differ and a string-exact diff would be impossible. The no-float rule and the verification strategy are the same decision — that's the loop closing.
SECTION 10

Seeing it run — and an honest status

Two PC-only tools let us watch the exact same firmware in action, without touching the embedded core.

The live viewer (live_view.cpp) — driving the firmware unchanged

This C++/OpenCV tool grabs webcam or video frames, converts them to grayscale, and hands the raw bytes to the identical degradation_update() that compiles for the Cortex-M0. What you see on screen is bit-for-bit what the chip would decide — which makes it the real tuning interface.

📹 webcam / video file cv::VideoCapture
↓
grayscale + force contiguous cvtColor → BGR2GRAY; clone() if row-padded, to honour the "packed row-major" contract
↓
⚙️ degradation_update() — the UNCHANGED embedded C core linked via extern "C" so the C++ build resolves the plain C symbol
↓
🖥️ overlay class-coloured border · 4×3 grid shading · per-feature bars with threshold ticks & pass/fail dots
extern "C" — the C++ ↔ C bridge C++ "mangles" function names (it encodes argument types into the symbol). C doesn't. Wrapping the header in extern "C" { … } tells the C++ compiler to look up the plain, un-mangled C name at link time — so the call resolves to the real firmware object with zero adapters and zero changes to the embedded code. The whole "same code on screen as on the chip" guarantee rests on this plus the packed-buffer check.

The batch harness (test_harness.c)

The only core-adjacent file allowed to use stdio.h. It reads headerless raw frames (just width×height bytes back-to-back, exactly what a camera DMAs into memory — produce them from any video with ffmpeg … -f rawvideo -vf "scale=320:240,format=gray"), runs the core, and writes a CSV of every feature so you can plot a metric collapsing as fog rolls in. It processes every 3rd frame, turning 30 fps into the firmware's ~10 Hz cadence.

The one-paragraph recap

A tiny, FPU-less, sub-2KB ARM chip must flag a degraded camera in real time. That single constraint forces integer-only, single-pass, shift-based code. One stride-4 pass turns each frame into ~9 cheap features; six of them feed a priority if-ladder (OBSTRUCTION → FROST → FOG → CLEAN) that reads the combination of features — because no single number separates the causes. Asymmetric hysteresis (5 in / 15 out) kills flicker. The design evolved from the brief's ratio scheme to simpler absolute thresholds, and a bit-exact Python oracle plus a 165-frame equivalence test prove the C is correct. PC, Python, and the live viewer all check out; the ARM-target verification is the honest next step.
